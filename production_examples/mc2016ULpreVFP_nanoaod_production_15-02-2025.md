# NanoAOD production campaign for 2016 preVFP MC

## Used configuration and setup:

* conditions: [`configuration/conditions.yaml`](../configuration/conditions.yaml)
* cmsdriver settings for nanoAOD: [`configuration/cmsdriver_nanoaod_specifics.yaml`](../configuration/cmsdriver_nanoaod_specifics.yaml)
* datasets: [`configuration/datasets_miniaod_mc_2016ULpreVFP.yaml`](../configuration/datasets_miniaod_mc_2016ULpreVFP.yaml)
* crab config template: [`crab_configuration/crab_template.py`](../crab_configuration/crab_template.py)
* CMSSW release: CMSSW_13_0_21

## Creation of CMSSW and Crab configs call

General call:

```bash
./create_configs.py --work-directory /ceph/$(whoami)/htt_mc_2016preVFP_crab_nanoaod_submission_15-02-2025_prodreleasev12/ \
  --datasets configuration/datasets_miniaod_mc_2016ULpreVFP.yaml \
  --conditions configuration/conditions.yaml \
  --cmsdriver configuration/cmsdriver_nanoaod_specifics.yaml \
  --nThreads 4 --numCores 4 \
  --maxMemoryMBperCore 750 --publication --splitting EventAwareLumiBased --unitsPerJob 500000 --maxJobRuntimeMin 1440
```

## Managing of crab tasks call

Initial calls will be presented in the following, which can be adapted further for resubmission supplemented with available resubmission options (see `./create_configs.py --help`).

```bash
./crab_manager.py --crab-config-pattern \
  /ceph/$(whoami)/htt_mc_2016preVFP_crab_nanoaod_submission_15-02-2025_prodreleasev12*/crabconfigs/*.py
```

### Adaptations during production campaign

Some of the datasets were initially produced with an inconsistent number of events. To do a more reliable iteration, these input datasets
were moved to T1_DE_KIT_Disk, and the tasks were re-submited with an adapted config, restricting them to run at KIT via `--siteWhitelist T1_DE_KIT`.
This includes running the jobs at nearby sites `T2_DE_DESY` and `T2_DE_RWTH`.

Some jobs needed resubmission with longer maximum runtime. This was done using additional options for better handling (`--rotation-mode [...] --resubmit-failed-immediately --sleep-duration 60`):

```bash
./crab_manager.py --crab-config-pattern \
  /ceph/$(whoami)/htt_mc_2016preVFP_crab_nanoaod_submission_15-02-2025_prodreleasev12*/crabconfigs/*.py \
  --nworkers 1 --rotation-mode --maxjobruntime 2750 --maxmemory 4000 --resubmit-failed-immediately --sleep-duration 60 
```

For some jobs, lumis were spread across files. In such cases, `EventAwareLumiBased` splitting might fail. Switching to `FileBased` splotting with 1 unit per job should fix such an issue. In such cases, adjust the runtime to ensure, that the increased amount of jobs will not stay too long in the queue.

In case the setup is on CERN lxplus, there are some tricks needed to keep the bash session alive. First it is good to have a ´tmux´ session open, from which you connect to CERN lxplus.
To keep the lxplus session alive for at least a week, you would need to renew the `kerberos` token by running the following script in background via `./renew_session.sh &`:

```bash
#!/usr/bin/env bash
while true
do
    kinit -R
    sleep 43200
done
```
